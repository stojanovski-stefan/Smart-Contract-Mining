# Replication Package

**Mining and Classifying Smart Contract Issues on GitHub**
**Repository:** [https://github.com/stojanovski-stefan/Smart-Contract-Mining](https://github.com/stojanovski-stefan/Smart-Contract-Mining)

---

## Overview

This project mines GitHub repositories written in Solidity to collect issues related to smart contracts. It then identifies fixing commits linked to those issues and classifies both issues and commits in a taxonomy. The pipeline produces a dataset of ~8,000 (as of May 2026) de-duplicated issues and ~1,800 fixing commits spanning 2018--2024, along with visualizations that characterize the data.

## Pipeline Description

### Step 1 -- Repository Discovery (`collect_repos.py`)

Queries the GitHub Search API for public Solidity repositories. De-duplicates by repository name.

- **Input:** GitHub API
- **Output:** `data/repos.json`

### Step 2 -- Issue Collection (`collect_issues.py`)

For each discovered repository, fetches issues whose title or body contains at least one of 12 smart-contract-related keywords (e.g., `reentrancy`, `overflow`, `solidity`, `vulnerability`). Work is partitioned by date range between two researchers so data can be collected in parallel.

- **Input:** `data/repos.json`
- **Output:** `data/issues_shard_{researcher}.jsonl`

### Step 3 -- Fixing Commit Collection (`collect_commits.py`)

For each collected issue, searches the repository's commit and pull request history for commits that reference the issue via closing keywords (e.g., "fixes #42", "closes #100").

- **Input:** `data/issues_shard_{researcher}.jsonl`
- **Output:** `data/commits_shard_{researcher}.jsonl`

### Step 4 -- Merge and De-duplicate (`merge_shards.py`)

Merges both researchers' shards into unified datasets, de-duplicating by `(repo_full_name, issue_number)` for issues and `(repo_full_name, sha)` for commits. Prints summary.

- **Input:** All shard files
- **Output:** `data/merged/issues_merged.jsonl`, `data/merged/commits_merged.jsonl`

### Step 5 -- Classification and Visualization (`classify_and_visualize.py`)

Applies rule-based heuristic classifiers to label each issue and commit, then generates eight charts.

**Issue classification:**
- **Issue type** (10 categories): security, bug, gas, feature, question, deployment, testing, maintenance, documentation, other
- **Vulnerability subtype** (8 categories, security issues only): reentrancy, overflow, access_control, front_running, dos, flash_loan, signature_replay, other_vuln

**Commit classification:**
- **Fix type** (8 categories): security_patch, bug_fix, test, refactor, feature, docs, maintenance, merge, other
- **Fix scope** (7 categories): contract_only, test_only, docs_only, contract_and_test, mixed, infrastructure, other

**Generated charts:**
1. `issue_type_distribution.png` -- Bar chart of issue type counts
2. `vuln_subtype_breakdown.png` -- Bar chart of vulnerability subtypes
3. `fix_type_distribution.png` -- Bar chart of fix type counts
4. `issues_over_time.png` -- Stacked area chart of issue volume by month
5. `fix_scope_file_types.png` -- Top 10 file extensions in fixing commits
6. `issue_fix_heatmap.png` -- Heatmap of issue type vs. fix type co-occurrence
7. `resolution_time_boxplot.png` -- Days-to-close by issue type
8. `keyword_frequency.png` -- Top keywords and labels

- **Input:** `data/merged/*.jsonl`
- **Output:** `data/classified/*.jsonl`, `data/figures/*.png`

## Replication Instructions

### Prerequisites

- A GitHub personal access token (classic) with `public_repo` scope

### Setup

```bash
git clone https://github.com/stojanovski-stefan/Smart-Contract-Mining.git
cd Smart-Contract-Mining/github_miner

python -m venv venv
source venv/bin/activate        # Mac/Linux
# venv\Scripts\activate         # Windows

pip install -r requirements.txt
```

Create a `.env` file in the `github_miner/` directory:

```
GITHUB_TOKEN=ghp_your_token_here
```

### Running the Pipeline
Full replication from scratch** (requires GitHub API access; takes several days due to rate limits)

```bash
# Step 1 -- Discover repositories
python collect_repos.py

# Step 2 -- Collect issues (run for each researcher or just one)
python collect_issues.py --researcher stefan
python collect_issues.py --researcher evan

# Step 3 -- Collect fixing commits
python collect_commits.py --researcher stefan
python collect_commits.py --researcher evan

# Step 4 -- Merge shards
python merge_shards.py

# Step 5 -- Classify and generate charts
python classify_and_visualize.py
```

## Configuration

All configurable parameters are in `config.py`:

- **Keywords:** 12 terms used to filter issues (reentrancy, overflow, gas, solidity, etc.)
- **Date range:** Monthly windows from 2018-01 to 2024-12
- **Repository filter:** `language:Solidity stars:>5 is:public`
- **Shard split:** Evan covers 2018-01 to 2021-06; Stefan covers 2021-07 to 2024-12
- **Closing keywords:** fixes, closes, resolves (and variants)
- **Contract file extensions:** `.sol`
